## Project Summary

AI-Browser 是一个基于 Electron 的多 AI 模型分屏浏览器。它让用户在同一个窗口里
同时对比 ChatGPT / Claude / Gemini / DeepSeek / 智谱清言 等 AI 的回答，通过广播
一次提问同步到所有主面板，并支持为同一 AI 打开多个彼此会话隔离的副本，以便
用多个账号并行工作、最大化每个账号的可用配额。

## Phase A — Foundation (Shared Infrastructure)

_Electron 壳、webview、持久化 session、CSP 剥离等基础设施。本阶段在项目初始化时已隐式完成，未经过 Trellis 正式验证。_

**What Phase A delivers:**
- Electron 28 主窗口 + 持久化 userData（`~/Documents/AI-Browser-Cache`）
- `persist:aisession` 默认分区用于 AI 站点登录持久化
- webRequest.onHeadersReceived 剥离 CSP / X-Frame-Options，允许 AI 站点在 webview 内加载
- 4 个主面板（ChatGPT / Claude / Gemini / DeepSeek）与统一广播框架

**Phase A verification milestone:**
_"Phase A is complete when I can open AI-Browser,看到 4 个 AI 站点并保持登录状态跨重启。"_（已满足，未经 Trellis 正式验证，待 `/trellis → *verify` 回补。）

## Phase B — 模型扩展与多账号并行

_面向小何的实际使用场景：扩展第 5 个 AI，并让任一 AI 可通过会话隔离的副本承载多账号。_

**Outcomes:**
- [x] **Outcome 1 — 小何在同一屏幕对比多个 AI，并用多个账号最大化使用量**
      — 新增智谱清言面板；任一 AI 可一键克隆独立 session 的副本；
      广播仅对主面板生效，副本排除。
      详见 [`docs/outcomes/outcome-01-zhipu-and-multi-account.md`](outcomes/outcome-01-zhipu-and-multi-account.md)
- [x] **Outcome 2 — 广播发送自动提交 + 代理配置 + 本地时间**
      — 修复广播注入后发送按钮未自动点击的问题（MouseEvent/PointerEvent 分发 + Enter 回退）；
      新增代理配置功能（HTTP/HTTPS/SOCKS4/SOCKS5，含认证）；
      状态栏新增本地时间时钟。
      详见 [`docs/outcomes/outcome-02-proxy-broadcast-fix.md`](outcomes/outcome-02-proxy-broadcast-fix.md)

**Independent outcomes in this phase (can build in any order):**
- Outcome 1 与 Outcome 2 均为 Phase B 独立 outcome，互不依赖。

**Phase B verification milestone:**
_"Phase B is complete when 小何 can：在 5 个主面板中广播同一问题、每个面板自动提交生成回复、为任一 AI 新增隔离副本、通过代理访问 AI 网站、状态栏显示本地时间。"_

## Phase C — 安全与反风控

_面向小何的实际痛点：多个 AI 面板同时访问 GPT/Claude 等网站时，因浏览器指纹相同被风控识别，需要每个面板拥有独立且自洽的伪造指纹。_

**Outcomes:**
- [x] **Outcome 3 — 指纹浏览器防护**
      — 为每个 webview 面板注入指纹伪装：Navigator/Screen/Canvas/WebGL/Audio/WebRTC API Hook；
      每个面板（含副本）拥有独立且参数自洽的指纹；一键开关控制。
      详见 [`docs/outcomes/outcome-03-fingerprint-browser.md`](outcomes/outcome-03-fingerprint-browser.md)

**Independent outcomes in this phase:**
- Outcome 3 不依赖其他 outcome，可独立构建和验证。

**Phase C verification milestone:**
_"Phase C is complete when 小何 can：启用指纹伪装后，GPT 和 Claude 无法通过浏览器指纹检测到异常；每个面板的 Canvas/WebGL 指纹相互独立；参数自洽无矛盾。"_

## Phase D — 用户体验优化

_面向小何的实际痛点：窗口缩小或面板较多时，工具栏拥挤、智谱（chatglm.cn）内容溢出，影响使用体验。_

**Outcomes:**
- [x] **Outcome 4 — 智谱面板响应式适配 + 全局响应式断点**
      — 新增 3 级响应式 CSS 断点（1200/900/700px），工具栏/标题栏/面板头部自适应缩小；
      智谱 webview 加载后自动注入响应式 CSS，图片等比缩放、代码块自动换行、表格可滚动。
      详见 [`docs/outcomes/outcome-04-responsive-zhipu.md`](outcomes/outcome-04-responsive-zhipu.md)

**Independent outcomes in this phase:**
- Outcome 4 不依赖其他 outcome，可独立构建和验证。

**Phase D verification milestone:**
_"Phase D is complete when 小何 can：在不同窗口宽度下正常使用所有功能；智谱面板在窄宽度下无内容溢出。"_

## Build Order — Why This Sequence

1. **先 Phase A**（已就绪）：没有 Electron 壳 + webview + 持久化 session，任何 AI 面板都无法工作。
2. **再 Phase B Outcome 1**：智谱只是第 5 个 AI 类型，复用 Phase A；多账号副本依赖 Electron `partition` 能力，也属于 Phase A 基础能力的扩展。
3. 因此 Outcome 1 可在 Phase A 完成后立即构建，无需其他 outcome 前置。
